## 2. 编码规范 (***通用部分，基本不用改***)

* **命名约定**:
    * 变量/函数: 小驼峰法 (camelCase)
    * 类/组件: 大驼峰法 (PascalCase)
* **代码风格**:
    * **强制格式化**: 所有提交的代码必须通过 Prettier 格式化。
* **注释**:
    * 所有公开的函数和组件都必须有 JSDoc 风格的注释。
* **错误处理**:
    * 禁止在代码中出现空的 `catch` 块。

## 3. 版本控制规范 (***通用部分，基本不用改***)

* **分支模型**:
    * 新功能分支命名: `feature/<feature-name>`
    * Bug 修复分支命名: `fix/<issue-number>-<description>`
* **提交信息 (Commit Message)**:
    * **必须遵循 Conventional Commits 规范**。
    * 格式: `<type>(<scope>): <subject>`
    * *Trae 指令：当你为我生成 commit message 建议时，必须严格遵循此格式。*

## 4. 测试规范 (***通用部分，基本不用改***)

* **测试要求**: 核心业务逻辑必须编写单元测试。
* **测试代码生成**: 遵循 **Arrange-Act-Assert (AAA)** 模式编写测试用例。
